Detalhe integração Jarvis-Linear

✅ INTEGRAÇÃO DETALHADA JARVIS CORE ↔ LINEAR (War Machine Kanban)
Versão 2.1 – Abril 2026

Aqui está a integração completa, prática e robusta entre o Jarvis Core (HITL Engine) e o Linear que você já usa hoje. O objetivo é transformar o Linear no frontend visual do Kanban da War Machine, enquanto o Jarvis atua como cérebro orquestrador (triagem, criação, atualização, análise de caos, auto-remediation, score e governança).

1. Visão da Integração (Arquitetura)
Fluxo Bidirecional:

Linear → Jarvis: Qualquer mudança no Linear (criação, update, comentário, move de estado) dispara webhook → Jarvis recebe, processa e reage (ex: inicia experimento de caos, atualiza SLO, cria ticket A2A).
Jarvis → Linear: Jarvis cria issues automaticamente, atualiza status, adiciona comentários com pensamento (Agent Activity), muda assignee, labels, priority, e gera post-mortems.
Componentes principais:

Linear API + Agent API (Linear tem suporte nativo para agents em 2026 — agente aparece como assignee, @mention e mostra “thinking”).
Webhooks do Linear (para eventos em tempo real).
n8n (na sua KVM) como camada intermediária (webhook receiver + orquestração).
Jarvis Core (Ollama local + agentes A2A) como decisores.
2. Métodos de Integração (do mais simples ao mais poderoso)
Método	Complexidade	O que permite	Recomendado para
Webhooks + n8n	Baixa	Receber eventos do Linear e enviar para Jarvis	Início rápido (recomendado agora)
Linear GraphQL API	Média	Criar, atualizar, ler issues, labels, projects	Jarvis criar issues automaticamente
Linear Agent API (nativo 2026)	Média-Alta	Jarvis aparece como agente oficial no Linear (com thinking steps)	Experiência mais fluida e profissional
MCP / Composio / StackOne	Alta	Integração via ferramentas prontas para agents	Futuro (quando escalar)
Recomendação atual: Comece com Webhooks + n8n + GraphQL API. Depois evolua para Linear Agent API para ter Jarvis como um “membro” visível no Linear.

3. Configuração Prática Passo a Passo
Passo 1: Criar API Key no Linear

Vá em Settings → API → Personal API Keys.
Crie uma key com escopo completo (ou mínimo: read, write, issues, comments).
Guarde em variável de ambiente na KVM (LINEAR_API_KEY).
Passo 2: Configurar Webhooks no Linear

Settings → Administration → API & Webhooks.
Crie webhook apontando para seu endpoint no n8n (ex: https://seu-dominio.ruptur.cloud/webhooks/linear).
Selecione resourceTypes: Issue, Comment, Project, Label.
Ative para o Team “War Machine”.
Passo 3: No n8n (sua KVM)

Crie workflow “Linear → Jarvis Webhook Receiver”.
Trigger: Webhook (POST).
Nodes:
Parse payload.
Decidir ação (ex: issue moved to “Chaos Review” → chamar agente Jarvis Chaos).
Chamar GraphQL da Linear para atualizar issue.
Envie eventos importantes para Jarvis via API interna (ex: /jarvis/event).
Passo 4: Jarvis Side (no core-hitl)

Crie agente especializado: Jarvis-War-Agent.
Ferramentas disponíveis para ele:
create_linear_issue(title, description, priority, labels, team)
update_linear_issue(id, status, assignee, labels, comment)
add_comment_with_thinking(issueId, thought, action)
get_issue_details(id)
Quando Jarvis decide criar um card (ex: detecção de conta legacy), ele chama a ferramenta acima.

5. Boas Práticas para esta Integração
Atomicidade: Use locks no Jarvis para evitar que dois agentes atualizem o mesmo issue ao mesmo tempo.
Thinking Visibility: Quando usar Agent API, Jarvis deve emitir “thought” em até 10 segundos (padrão Linear Agent).
Score de Qualidade: Todo issue fechado recebe score (0-100) calculado pelo Jarvis (segurança + resiliência + impacto receita).
Labels padronizadas: Mantenha consistência (ga-identity, chaos-engineering, etc.).
Backup: Mantenha sincronização eventual com GitHub (ruptur/war-kanban como source of truth secundário).
Segurança: Nunca exponha Linear API Key diretamente no prompt do Ollama — use tool calling com secrets gerenciadas pelo Jarvis.
